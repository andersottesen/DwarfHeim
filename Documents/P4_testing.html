Tester Audience

Our test audience consists of both potential gamers and our client. This audience will also be a part of the end user-group of the application. We tested on two students who play a lot of games. The web application was also tested by the client, during a meeting. We did not have access to any potential investors, so we were not able to test on them.

Testing Document

To test our site we decided to do a sit-down test with both potential gamers and the client. For potential gamers we wanted to test two things; the page layout and the feel of the page. To test the layout, we gave the users some small tasks, such that they could navigate through the web application. Afterwards we asked them some questions about the look and feel of the page. The questions we asked were:

Test tasks:

  1. Find Eirik Fosse's phone number
  2. Look at all the pictures
  3. Watch a video
  4. Read about the latest updates in the game

Follow-up questions:

  1. How would you describe the mood/feel of the page? (dark, happy, dramatic etc. )
  2. Do you want to play the game?
  3. What do you think about the balance between information and sale?

We also wanted some concrete answers and feedback from our client, and we asked them the following:

  1. Do you think the web page is logical? Do you have any problem with navigating through the web application?
  2. What do you think about the graphical design of the web page?
  3. What do you think about the web site's atmosphere, is it too dark?
  4. How is the balance between informational content and the marketing of the site?

Testing Platforms

Both mobile devices and browsers on computers were used as testing platforms when we conducted the user tests. We tested the following browsers; Google Chrome, Safari, Firefox, Internet Explorer. This was to see how the GUI and layout turned out on the different devices.

Feedback Summary

Potential gamers

Test person 1

  • "The landing-page is to big. There should not be a need to scroll. Also the logo should be a link."
  • Finding the phone number was easy.
  • Finding and looking through the pictures was no problem.
  • Did not like that the heading-fonts were different from the navigationbar-fonts.
  • Needed more time and information before he could say anything about wanting to play the game.

Test person 2

  • Finding the phone number was easy.
  • Navigated through “The game” to look at the pictures.
  • Found the media-page, and this was easy.
  • Did not have any problems with finding “the news”
  • Navigated without problem to “the game”
  • "The background is a bit boring". The test person wanted more patterns or more decorations on the background on the sides, or maybe another color.
  • The test person though it was too much space between objects on the web site.
  • "The header-picture is really cool! This sets the mood and atmosphere."
  • Thought it was a lack of information on the site. Could have been more interesting with more content on the web site. "Like this site for instance: wow-one.com"

Client

The customer was pleased with the site we showed them. They liked how the site looked, and that it wasn't too dark and serious, although the colours and sidebars were a bit boring. Navigating through the site was no problem, and they found all the information they were asked fairly quickly. One page they did not like that much was the page with information about the game (dwarfheim.php). The image should be bigger, and the layout of the information was not appealing enough. The interviews page on the other hand was really good, and they like how there is one page with every team member with contact info, and that each person has their own interview page.

All in all they really like the page, and with some adjustments to the content they provided (text, images, etc.) they will soon host the site on their own server.

Our Response

The tests revealed that the lack of information on the page is a big problem, something we already knew. We did not receive more information from the client, most likely due to the fact that their project is newly started and that there just are not any information available at this time. This however, is easy to fix once the information exists. We implemented the minor changes that the test audience suggested right away. These are described in detail in out changes document. Since we were not supposed to do any huge changes after the testing, except for the eventual critical flaws, we chose not to implement the major changes, such as new graphics for the side-bars.

We made an agreement with the customer that we will not make any changes to the site based on the test results we got from them. They will instead make the necessary changes in layout and colours themselves, before they publish it to their own server. This was favourable for both parties, since it gives them freedom to test different layouts and colours.

Self Reflection

In addition to learn the general structure and use of HTML, CSS and JavaScript,this project taught us that implementing a web application like this takes a lot of coordination and planning. It is a puzzle to make sure look and feel matches. Having a good and comprehensive design document is essential. Also spending time together as a team is important. When the team meets, problems that arises can be solved fast and new ideas can be incorporated in the site.

We all felt it was challenging to create a proper web application without back-end. We had an actual client in need of a functional page, and it would have been nice to be able to complete it, and add other functionality that requires back-end. On the other hand, we realise that completing a functional web application with both front- and back-end would be a lot of work, and that the project then would become too extensive.

We had a lot of fun to actually create a web application from scratch, and we enjoyed this project quite a lot. Hopefully our client will be able to use our project as a starting point to their own web sites.
